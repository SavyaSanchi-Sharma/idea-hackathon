# ZombieHunter — Zombie & Shadow API Discovery for Public Sector Banks

## Problem Statement

This project addresses **PS: Zombie & Shadow API Discovery for Public Sector Banks**. ZombieHunter continuously inventories a bank's API estate, classifies every endpoint as `active`, `deprecated`, or `orphaned` (with `zombie` and `shadow` flags layered on top), scores its risk on a 0–100 scale, surfaces OWASP API Top-10 findings per endpoint, and delivers the result to an investigator console designed as a forensic instrument — not a generic dashboard.

## Tech Stack

**AI Engine (`ai_engine/`)**
- Python 3.11
- FastAPI + Uvicorn (inference API, WebSocket scan stream)
- scikit-learn — Random Forest classifier (3-class lifecycle), Random Forest regressor (risk 0–100), Isolation Forest (sequence anomaly)
- joblib (model artifacts), Pandas + NumPy (data plane)
- Deterministic rule classifier cross-checked against the ML pass — disagreements flow to a Review Queue

**Frontend (`frontend/`)**
- React 18 + TypeScript + Vite
- TailwindCSS with custom design tokens (the "STRATA" system — see `design/`)
- D3 (`d3-zoom`, `d3-selection`) for the stratigraphic landscape graph
- Recharts (sparklines, sequence detail charts), Framer Motion (scan / decay animations)
- TanStack Query (server state), Zustand (UI state), React Router

**Data**
- 1,920 synthetic banking endpoints generated from ~50 OpenAPI specs (45 from APIs.guru + Stripe, Plaid, Adyen, PayPal as structural seeds)
- 30-day per-endpoint telemetry sequences (57,600 rows)
- CVE join table covering 13 (runtime, version) pairs across Spring Boot / Node.js / Python / Go / .NET
- Generator is deterministic on `SEED=42` — see `ai_engine/train/DATA.md`

## Dataset

All data is 100% synthetic, generated by `ai_engine/train/data.ipynb`. It simulates a mid-size PSB's API estate:

- **1,920 endpoints** drawn from 50 real OpenAPI specs, overlaid with banking paths from a 45-template lexicon (`/v{1,2,3}/accounts/{accountId}/balance`, `/upi/collect`, `/v2/loans/applications/{id}/disburse`, …) and 10 services (`core-banking`, `payments`, `cards`, `loans`, `kyc`, `wealth`, `forex`, `notifications`, `audit`, `aa-bridge`).
- Each endpoint is sampled into one of 5 scenarios with weights `(active 0.50, deprecated 0.15, orphaned_quiet 0.10, zombie 0.15, shadow 0.10)`. The scenario drives the feature distribution.
- **30-day sequences** per endpoint (57,600 rows) with realistic per-scenario trajectories. 10% of endpoints receive an injected behavior shift (3× spike or 10× drop) to train the anomaly detector.
- **CVE overlay** — every runtime+version joins to a curated CVE row, so each endpoint carries a `max_cvss` ∈ [0, 10] used by the risk model and OWASP rule engine.

No real bank data was used. All endpoint IDs, account numbers, and customer references in fixture content are fabricated.

## Model Performance (on Synthetic Test Set)

| Model | Task | Notes |
|---|---|---|
| Classifier (Random Forest) | 3-class lifecycle (`active` / `deprecated` / `orphaned`) | High separability by design — `in_registry`, `owner_present`, `last_deploy_days`, `call_count_7d` each linearly separate at least one class pair (see `DATA.md` §5.5). |
| Regressor (Random Forest) | Risk score 0–100 | `risk_score` is a deterministic function of the same features plus `N(0,5)` noise (see `DATA.md` §4.7). Proves the inference path works end-to-end, not generalization to real-world risk. |
| Anomaly detector (Isolation Forest) | Behavior-shift detection on 14 sequence features | Trained on 30-day trajectories; ~10% of endpoints have injected shifts. Surfaced as a binary flag + score; recall is tunable via the contamination parameter. |
| Rule classifier | Deterministic ground-truth pass | Run alongside ML classifier; disagreements set `needs_review=1` and flow to the Review Queue surfaced in the UI. |
| OWASP rule engine | API Top-10 findings (API1, API2, API4, API8, API9, API10) | Pure rules, audit-defensible. Each category has a stated predicate (`DATA.md` §4.8). |

Note: These results are on synthetic data. Performance on real PSB telemetry would require re-training and re-feature-engineering.

## Known Limitations

- **Synthetic-only.** Trained against generated data; real PSB telemetry would require re-feature-engineering and re-training against labels from the bank's own incident history.
- **Batch ingestion in the POC.** Discovery is mocked from CSV at startup. The production Rust ingest plane is designed but not yet built.
- **No live SLM in this build.** Threat-narrative paragraphs are template-filled from structured signals. Phase 2 swaps in a locally-hosted small language model — bank traffic patterns must not leave the perimeter, ruling out hosted APIs.
- **OWASP coverage is 6 of 10 categories.** API3, API5, API6, API7 need response-payload signals not currently simulated.
- **No write-path automation.** Graduated response (quarantine / rate-limit / block) is designed but unimplemented — UI surfaces the recommendation; a human operator executes it.
- **No multi-tenant isolation in the POC.** A real SaaS deployment across PSBs needs per-bank data and model isolation; the inference cache is currently single-tenant.
